## 1. What this is

A Persian, **RTL**, **mobile-first** property-management web app for a real-estate office (`املاک ایران زمین` / "DealEstate"). Non-technical office staff use it to manage property files, people, contracts, and buyer/renter requests.

- **Frontend**: React (TypeScript) in `frontend/`.
- **Backend**: Django + Django REST Framework in `backend/`, built in the **HackSoft Django Styleguide** (https://github.com/HackSoftware/Django-Styleguide).
- **DB**: MySQL, already running in Docker at `127.0.0.1`, user `root`, password `abcd@1234`, database `iranzamin`.
- **Python packaging**: **uv** only (no pip/poetry).
- **Design source of truth**: `design/` (do not edit). Build the UI to match it exactly.

The full app flow and every screen live in `design/DealEstate/DealEstate.dc.html`. The design system (tokens, components, rules) is in `design/DealEstate/_ds/.../` — read `readme.md` there before touching any UI.

## 2. Golden rules (non-negotiable)

1. **Design fidelity is mandatory.** Frontend must be pixel-faithful to `design/`. Same layout, spacing, colors, radii, shadows, copy, icons, flow. Use the design tokens — never hardcode hex/px that a token already defines.
2. **After every frontend change**, run the app, open it in a browser, take a **screenshot**, and verify it matches the design. If you can see the screenshot, confirm it visually; note any diff and fix before moving on. Test mobile viewport (≤480px) AND desktop (≥920px sidebar layout).
3. **Backend follows HackSoft Styleguide** (see §5). Business logic in `services.py`, reads in `selectors.py`, thin models, thin APIs. No fat models, no DRF ViewSets/routers, no logic in serializers.
4. **Tests are required for everything implemented or changed** (see §7). Include **user-behavioral / edge-case** tests, not just happy path.
5. **Commit after each edit / logical unit of work** (see §8). Good message with a conventional prefix. **Never** mention an AI, coding agent, Claude, or "generated by" anywhere in commits or code.
6. **Persian everywhere in the UI**: Persian digits (۰۱۲۳۴۵۶۷۸۹), `٬` thousands / `٫` decimal, RTL, Vazirmatn, no emoji in the product UI. Status vocab is canonical: `خالی`=available (green), `پر`=occupied (red).
7. **uv for all Python**, MySQL (via **PyMySQL**) for DB. Don't swap the stack.
8. **Code style: simple, optimized, human-readable.** Write the plainest code that does the job well. Favor clarity over cleverness — someone skimming should understand it instantly. Follow **DRY** (no copy-paste; extract shared logic) and **SOLID** (single-responsibility, small focused units, depend on abstractions). Short functions, clear names, no dead code, no premature abstraction. Simple beats clever every time.

## 4. Domain model (from the design)

Entities and their key fields, derived from the wizard and screens in `DealEstate.dc.html`. Model these in Django; names below are the Persian labels the UI uses.

- **User / Agent (مشاور املاک)** — auth by mobile number + password. Profile: name, phone, notification + dark-mode settings.
- **Region (منطقه)** — neighborhoods; addable inline via the Picker ("افزودن منطقه جدید").
- **Person (شخص)** — a `مالک` (owner) or `مشتری` (customer/renter/buyer). Fields: first name, last name, phone, national id (کد ملی), birth date. Linked to properties/contracts/requests.
- **Property / File (ملک / فایل)** — the core entity. Type is one of `آپارتمان` (apartment), `کلنگی` (kalnagi/teardown), `زمین` (land), plus `تجاری`/`اداری`/`ویلا` from the brief. Common: region, address, plak (پلاک). Type-specific:
  - Apartment: floor (طبقه), unit (واحد), area (متراژ), beds (تعداد خواب), amenities (امکانات), cabinet material (جنس کابینت), build year (سال ساخت), storage/انباری (+سندی +area), تبدیل.
  - Kalnagi: area, عقب‌نشینی (+desc), تعداد بر, گذر کوچه, تعداد طبقات, حیاط (+area).
  - Land: area, عقب‌نشینی (+desc), تعداد بر, گذر کوچه.
  - **Deal types (چند انتخابی)**: `فروش` (sale: price-per-meter + total), `اجاره` (rent: deposit پول پیش + monthly اجاره), `رهن کامل` (full mortgage: پول رهن). Land is sale-only.
  - **Status**: `خالی` (available) / `پر` (occupied). Occupied → start/end dates (Jalali) + tenant (مستأجر).
  - Owner (مالک), photos, video, gallery.
- **Contract (قرارداد)** — links a property + parties (seller/owner + buyer/renter), type (فروش/اجاره/رهن), start/end dates (Jalali), amount(s), contract image, notes. Registering a contract updates the property's owner/tenant status.
- **Request (درخواست)** — a customer looking for a property. Customer (existing or quick-add), type (rent/mortgage vs buy), constraints: persons count, beds, needs (نیازها), preferred floor, area, build year, max deposit / max rent / budget, deadline (مهلت). Matching files are auto-suggested.

Prices are in **Toman**, stored as integers (no decimals); format for display on the frontend.

Dates in the UI are **Jalali (Shamsi)**. Store canonical dates and convert at the edges — use a Jalali library on the frontend (and `django-jalali` or explicit conversion on backend if needed for storage/queries).

## 5. Backend — HackSoft Django Styleguide

If unfamiliar, read the styleguide repo first. Apply it strictly.

### App structure
Each Django app:
```
apps/<app>/
├── models.py       # thin models: fields, properties, clean(), Meta
├── services.py     # ALL create/update/delete + business logic (functions)
├── selectors.py    # ALL reads/queries (functions returning querysets/objects)
├── apis.py         # thin DRF APIViews, one per action
├── urls.py         # explicit url patterns, named
├── filters.py      # django-filter filtersets (list endpoints)
├── admin.py
└── tests/
    ├── services/test_*.py
    ├── selectors/test_*.py
    └── apis/test_*.py
```

### Rules
- **Models are thin.** Fields, `Meta`, `__str__`, computed `@property`, and `clean()` for invariants. No business logic, no queries. Provide a `BaseModel` with `created_at`/`updated_at` and inherit it.
- **Services** hold all mutations and business logic. Plain functions, fully type-hinted, keyword args. Each service calls `full_clean()` before `save()` and wraps multi-step writes in `transaction.atomic()`. Example: `property_create(*, agent, type, region, ...) -> Property`.
- **Selectors** hold all reads. Functions like `property_list(*, filters=None) -> QuerySet[Property]`, `property_get(*, property_id) -> Property`.
- **APIs are thin.** One `APIView` per action (`PropertyListApi`, `PropertyCreateApi`, `PropertyDetailApi`, ...). Define `InputSerializer` / `OutputSerializer` **nested inside** the API class. The view validates input, calls a service or selector, serializes output. **No** ViewSets, **no** routers, **no** business logic or queries in serializers.
- **Errors**: use a custom `ApplicationError` and DRF custom exception handler; return structured error responses.
- **URLs**: explicit, named, nested per app; include under a versioned prefix `/api/`.
- **Settings**: split (`config/settings/base.py`, `local.py`, `test.py`, `production.py`). Read secrets/DB from env via `django-environ`. Never hardcode the DB password in committed code — put it in `backend/.env` (gitignored) and document it in `.env.example`.

### Stack / deps (via uv)
`django`, `djangorestframework`, `django-cors-headers`, `django-filter`, `django-environ`, `PyMySQL` (MySQL adapter), `djangorestframework-simplejwt` (auth), `django-jalali` (if storing Jalali). Dev: `pytest`, `pytest-django`, `factory_boy`, `coverage`/`pytest-cov`, `ruff`, `mypy`.

Use **PyMySQL** as the MySQL driver (pure-Python, no system build deps). Install it as `MySQLdb` at the top of `backend/config/__init__.py`:
```python
import pymysql
pymysql.install_as_MySQLdb()
```

## 6. Frontend — React

- **Vite + React + TypeScript**. RTL (`dir="rtl"`), Vazirmatn font, mobile-first.
- **Import the design tokens/CSS** from the design system (`design/DealEstate/_ds/.../tokens/*.css` + `styles.css`) rather than reinventing them — copy them into `frontend/src/styles/` (or reference) and use the CSS variables (`--color-primary`, `--radius-lg`, `--space-4`, ...). Never hardcode a value a token defines.
- **Layout**: single column capped at ~480px on mobile with a 64px bottom nav; at ≥920px switch to the fixed right-side sidebar (RTL) and hide the bottom nav — exactly as `DealEstate.dc.html` does.
- **Data**: TanStack Query (React Query) + a typed `axios` client. **Routing**: React Router. Keep screen structure/flow identical to the design (Login → Dashboard → Files/Properties → Detail → Add-wizard; Persons; Contracts; Requests; Profile).
- **Persian**: helper to convert digits to Persian and format Toman with `٬`. All numerals, prices, areas, counts, dates, phones render in Persian digits. Jalali dates via a library (e.g. `jalaali-js` / `react-multi-date-picker`).
- **No emoji** in product UI. Icons are **Lucide** outline, stroke-width 2, `currentColor` (the greeting `👋` in the dashboard header is the one intentional exception already in the design).
- **Reusable primitives** mirror the design system components: Button, IconButton, Input, Select, Switch, Badge, Card, Avatar, StatCard, PropertyCard, Tabs, BottomNav — plus Picker, JDate, Uploader from the wizard.

## 7. Testing

Coverage is a requirement, not an afterthought. For **everything you implement or change**, add/extend tests.

**Backend** (pytest + factory_boy):
- `services/` tests: business rules, validation, side effects (e.g. registering a contract flips property status and updates owner/tenant), `transaction.atomic` rollback on failure.
- `selectors/` tests: filtering, search, ordering, permissions.
- `apis/` tests: status codes, auth required, input validation errors, output shape.
- **User-behavioral / edge cases**: empty inputs, invalid Persian/English digits, huge prices, missing owner, occupied-without-dates, land with rent (should be rejected), duplicate phone, unauthorized access, pagination boundaries. Cover the sad paths, not just the happy one.
- Use factories for all model creation; keep tests isolated (DB per test).

**Frontend**:
- Component tests (Vitest + React Testing Library) for primitives and screens: rendering, Persian-digit formatting, RTL, validation messages, empty states (`ملکی با این مشخصات یافت نشد.`).
- Behavioral/e2e (Playwright): full flows — login, add-property wizard step-by-step (each property type branch), create contract, create request, search/filter. Assert on user-visible behavior and edge cases (back navigation, invalid form, network error toast).
- Run the visual verification in §6 as part of "done".

Never mark a task complete with failing or skipped tests. If something is skipped, say so explicitly.

## 8. Commits

- Commit after each logical edit/unit of work — small, focused commits.
- **Conventional Commits** prefix: `feat:`, `fix:`, `refactor:`, `test:`, `chore:`, `style:`, `docs:`, `build:`.
- Message describes what changed and why, in plain terms. Body only when the "why" isn't obvious.
- **Never** reference an AI, agent, assistant, Claude, "co-authored-by" a bot, or "generated by". Write as a normal human developer.
- If on `main`, prefer a feature branch for multi-commit work; otherwise commit directly per the user's flow.
- Only `git push` / open PRs when the user asks.

Examples:
```
feat: add property list api and selector
feat(frontend): build dashboard stat cards to match design
fix: reject rent deal type on land properties
test: cover contract registration updating property status
```

## 9. Workflow for a typical feature

1. Read the relevant screen in `design/DealEstate/DealEstate.dc.html` and the design-system rules.
2. **Backend**: model → migration → selectors → services → apis → urls → tests. Run `uv run pytest`.
3. **Frontend**: types/client → component(s) using design tokens → wire to API → tests.
4. **Verify visually** in the browser + screenshot (mobile + desktop). Fix diffs.
5. Run backend + frontend tests. All green.
6. **Commit** with a proper prefixed message (no AI mention) and push it.
